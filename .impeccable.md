# .impeccable — HADAL Design & Production Standards

> The rules that make HADAL feel institutional, not amateur.
> Read this before designing, prompting, or building any visual element.

---

## Design Principles

1. **Green Fallout is law.** Lemon-green (#C4FF2C) on near-black (#060800). No exceptions. Every pixel reinforces the CRT military terminal aesthetic. If it doesn't look like it belongs on a 1980s command center screen, it doesn't belong in HADAL.

2. **Density over decoration.** HADAL is a Bloomberg Terminal for threat intelligence. Every square pixel should carry data or context. Whitespace is acceptable only when it improves scan speed — never for "breathing room" or visual comfort.

3. **Typography is hierarchy.** Rajdhani for headers (authority), Teko for big numbers (impact), Share Tech Mono for data (precision), Cormorant Garamond for rare editorial moments (gravitas). Never mix roles. Never introduce new fonts.

4. **Motion is signal, not style.** Animations exist only to convey state: sonar sweep = active scanning, blink = live status, pulse = alert. If an animation doesn't communicate system state, remove it. No entrance animations, no hover flourishes, no loading spinners with personality.

5. **Everything is a panel.** The UI is a tiled command interface. Sharp corners (0px radius), thin borders (`1px solid rgba(196,255,44,.15)`), dark card backgrounds. Panels can be resized (sandbox mode) but never overlap, float, or cast shadows.

6. **Small mark, big world.** Logos and brand marks earn authority through restraint, not size or complexity. One shape, small in frame, dark environment, white or single-color mark. Negative space is the design. Never prompt a logo with gradient, color, and texture — that's noise. Define the world the mark lives in first, then place the mark modestly inside it. The brands that feel institutional never shout.

## Technical Constraints

### Canvas Rendering
- All custom visualizations use Canvas 2D API — no D3, no chart libraries, no SVG for animated elements
- Canvas hooks follow the pattern: `use[Name]` in `src/canvas/`, return a `ref` to attach to a `<canvas>` element
- Use `requestAnimationFrame` for animation loops, clean up on unmount
- Canvas dimensions are explicit (e.g., globe: 420x420, sonar: 140x140) — no responsive canvas scaling

### Color Application
- Primary green (`--g`) at full opacity only for key text and active indicators
- All other green usage must use opacity variants (`--g9` through `--g07`)
- Warning/alert color is amber (`--warn: rgba(255,140,0,.9)`) — reserved for threats and Iran
- Background layers: `--bg` (base) → `--bg1` (cards) → `--bg2` (elevated panels)

### CRT Effects Stack
- Layer order (bottom to top): background → content → grain overlay → scanlines → vignette
- Grain: SVG turbulence filter at 0.5 opacity
- Scanlines: 2px transparent + 2px black at 0.07 opacity, repeating
- Vignette: radial gradient darkening edges
- These are global — never apply per-component

### Data Display
- Numbers use Teko (`--C2`) at large sizes for dashboard impact
- Tables use Share Tech Mono (`--MONO`) for alignment and readability
- Status indicators: small colored dots with blink animation for live state
- Severity uses a 5-level scale: CRITICAL → HIGH → ELEVATED → GUARDED → LOW

### Particle & Ambient Animation
- For animated backgrounds, hero depth, or ambient effects: define shape preset → color → density → interaction behavior BEFORE coding
- Implementation priority: pure Canvas API first (zero dependencies), CSS-only for non-interactive ambient, tsparticles/particles.js only if interaction is required
- Reference tool for visual config: particles.casberry.in (exportable presets)
- HADAL candidates: sonar sweep particles, feed event burst effects, map marker spawn animations
